# Applications collection

A new `applications` collection parallel to `packages`. An application is a ONCE-compatible web app (Docker container, HTTP on :80, `/up` healthcheck, persistent data in `/storage`). The `once` package already exists in the marketplace and is what users use to deploy these.

## Decisions

- **Schema**: same as `packages` plus a required, unique `docker_image` field (registry + repo + tag).
- **Submission input**: `{ github_url, docker_image, description, tags }`. `name` is derived from `github_url` (same `parseGithubName` as packages).
- **Docker image validation**: public registries only. Blocking, on submit. HEAD `/v2/<repo>/manifests/<tag>` anonymously (Docker Hub token fetched anonymously when needed). Reject if not reachable.
- **Re-validation**: none. Nightly cron stays GitHub-only.
- **Protected fields**: same set as packages, plus `docker_image`.
- **Approval / rebuild dispatch**: identical model to packages (`dispatchRebuild` on approved create, any update, approved delete, nightly cron).
- **Routes**: parallel to packages.

## Work

### 1. Migration

`pocketbase/pb_migrations/<ts>_init_applications.js` — clone `1714000000_init_packages.js`:

- Collection name: `applications`.
- Add field `docker_image` (text, required, min length 1).
- Add unique index on `docker_image`.
- Same list/view/create/update/delete rules as `packages` (approved OR submitter; create requires auth; update/delete restricted to submitter or superuser per existing model).

### 2. Hooks

`pocketbase/pb_hooks/utils.js`:

- Add `validateDockerImage(image)`:
  - Parse `[registry/]repo[:tag]`. Default registry `registry-1.docker.io`, default tag `latest`. Normalize single-segment Docker Hub repos to `library/<repo>`.
  - For Docker Hub, fetch an anonymous token from `https://auth.docker.io/token?service=registry.docker.io&scope=repository:<repo>:pull`.
  - HEAD `https://<registry>/v2/<repo>/manifests/<tag>` with `Accept: application/vnd.oci.image.manifest.v1+json, application/vnd.docker.distribution.manifest.v2+json` and the token if obtained.
  - Throw a `BadRequestError` with a clear message on non-2xx.
- Add `"docker_image"` to `PROTECTED_FIELDS`.

`pocketbase/pb_hooks/applications.pb.js` — mirror `packages.pb.js`:

- `onRecordCreateRequest("applications")`: derive `name` from `github_url`, force `status="pending"`, `stars=0`, `submitter=e.auth.id`, then `validateDockerImage(e.record.get("docker_image"))` (blocking).
- `onRecordAfterCreateSuccess("applications")`: GitHub enrichment (description, stars, default_branch, pushed_at, og_image), then `dispatchRebuild` if approved.
- `onRecordUpdateRequest("applications")`: revert `PROTECTED_FIELDS` from `original()` for non-superusers.
- `onRecordAfterUpdateSuccess` / `onRecordAfterDeleteSuccess`: same dispatch rules as packages.
- Nightly cron: refresh GitHub stats for applications too (extend the existing cron or add a parallel one).
- Remember the JSVM scope rule: `require("./utils.js")` *inside* each callback.

### 3. Frontend

New pages under `web/src/pages/`:

- `applications/index.astro` — list (mirror of packages index).
- `applications/[id].astro` — detail. Show `docker_image` with a copy-to-clipboard button and a short "Deploy with the `once` package" hint linking to the once package page.
- `applications/submit.astro` — submission form with `github_url`, `docker_image`, `description`, `tags`.
- `my/applications.astro` — submitter's own applications (mirror of `my/packages`).

Extract shared list/card components from the packages pages where it's a clean lift; otherwise duplicate and revisit later. The submit form surfaces the validation error from the hook inline.

### 4. Navigation

Add an "Applications" entry alongside "Packages" in the site nav and on the home page.

### 5. Rebuild workflow

No change to `.github/workflows/rebuild.yml` — it rebuilds the whole site. Hooks fire `dispatchRebuild()` on the same triggers as packages.

## Out of scope

- Private registry auth.
- Image re-validation over time.
- Any work on the `once` package itself.
- Actual deploy-from-UI flow (the detail page only points users to `once`).
