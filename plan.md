# bigconfig marketplace — plan

A "Made with BigConfig" website: a curated, public index of GitHub repos that
are BigConfig packages (Clojure projects built on `big-config`, with a `bb.edn`
generated by an agent). Inspired by https://www.madewithpocketbase.com/.
Visitors browse and search; signed-in users submit and edit their own packages.

## Stack

- **PocketBase** — auth (Google SSO), SQLite DB, admin UI, JS hooks, static
  file server. Single Go binary on one host.
- **Astro** — static site generator. Build output written to
  `pocketbase/pb_public/`, served same-origin by PocketBase. No SSR.
- **Tailwind CSS** — styling, matched to bigconfig.it/marketplace (see
  "Look & feel").
- **PocketBase JS SDK** — runs in the browser; handles all dynamic behavior
  (auth, submit, edit, list).

Single origin: PocketBase serves both `/api/*` and the static site `/*`. No
CORS, no separate hosting.

## Data model

**`users`** — built-in PocketBase `users` collection, Google OAuth2 enabled in
Admin → Settings → Auth providers. Name and avatar come from the OAuth profile.

**`packages`**

| field            | type                     | notes                                              |
|------------------|--------------------------|----------------------------------------------------|
| `github_url`     | url                      | unique; canonical key                              |
| `name`           | text                     | `owner/repo`, derived from URL; not user-editable  |
| `description`    | text                     | user-editable; pre-filled from GitHub on submit    |
| `tags`           | select (multi, curated)  | see taxonomy below                                 |
| `submitter`      | relation → users         | auto-set to `@request.auth.id`                     |
| `status`         | select                   | `pending` / `approved` / `rejected`                |
| `stars`          | number                   | from GitHub; refreshed nightly                     |
| `default_branch` | text                     | from GitHub                                        |
| `pushed_at`      | date                     | from GitHub; used for "last updated" sort          |
| `og_image`       | url                      | GitHub OG image; for card thumbnails               |

### Access rules

- **List / View**: `status = "approved"` (public). Submitter can also see own
  pending rows.
- **Create**: `@request.auth.id != ""`. Hook forces `status = "pending"` and
  sets `submitter`.
- **Update**: `@request.auth.id = submitter.id`. Protected fields
  (`status`, `stars`, `name`, `github_url`, `submitter`, GitHub-derived fields)
  are blocked via hook.
- **Delete**: same as update, plus admins.

Admins moderate (approve/reject) via the built-in PB admin UI at `/_/`.

## PocketBase hooks (`pb_hooks/`)

- `packages.onRecordBeforeCreateRequest` — derive `name` from `github_url`,
  force `status = "pending"`, set `submitter` to authed user.
- `packages.onRecordAfterCreateSuccess` — fetch
  `https://api.github.com/repos/{name}` and populate `description` (if blank),
  `stars`, `default_branch`, `pushed_at`, `og_image`.
- `packages.onRecordBeforeUpdateRequest` — strip protected fields from
  non-admin updates.
- `cronAdd("github-refresh", "0 3 * * *", ...)` — nightly refresh of GitHub
  metadata for all approved packages.
- `packages.onRecordAfterUpdateSuccess` /
  `packages.onRecordAfterCreateSuccess` /
  `packages.onRecordAfterDeleteSuccess` — when the change affects the public
  listing (record is or was `approved`), POST `repository_dispatch` to
  GitHub to trigger a site rebuild. See "Rebuild pipeline" below.

## Astro routes

All routes are statically generated. Public listing pages fetch from PB at
build time; auth-gated pages are static shells with client-side logic.

- `/` — landing + most-recent packages grid. Pre-rendered with the latest N
  approved packages fetched from PB at build time.
- `/packages` — full grid with tag filters and text search. The full set of
  approved packages is fetched at build time and embedded as JSON; filtering
  and search run over it client-side.
- `/packages/[name]` — package detail. Generated via `getStaticPaths` that
  fetches all approved packages from PB; one HTML file per package.
- `/submit` — submit form. Static shell; auth guard runs client-side, redirects
  to `/login` if not authed.
- `/me` — submitter's own packages, client-rendered.
- `/me/edit/[id]` — edit form, client-rendered.
- `/login` — Google SSO button →
  `pb.collection('users').authWithOAuth2({ provider: 'google' })`.

Listing and detail pages are SSG with a build-time fetch from PB, so each
package gets its own indexable HTML file. Freshness is maintained by
triggering a rebuild whenever an approved record changes — see "Rebuild
pipeline" below.

## Auth

- Google OAuth2 via PocketBase's built-in provider. Requires a Google Cloud
  OAuth client; client ID/secret pasted into PB admin settings.
- Redirect URL: `https://<host>/api/oauth2-redirect`.
- Login flow runs entirely in the browser through the PB JS SDK; no Astro
  endpoints involved.

## Build / dev / prod

- **Dev**: `web/` runs `astro dev` on :4321, with a PB instance on :8090.
  `src/lib/pb.ts` reads `import.meta.env.PUBLIC_PB_URL` so dev points at
  `http://localhost:8090` and prod uses `''` (same origin). Build-time
  fetches in dev hit `http://localhost:8090`.
- **Build**: `cd web && npm run build` reads `PUBLIC_PB_URL` (prod URL in
  CI), fetches approved packages, and writes static output to
  `../pocketbase/pb_public`.
- **Prod**: hosting is handled by the user. PB serves the API and the static
  site on one port from one binary.

## Rebuild pipeline

Listing freshness is achieved by rebuilding the site whenever the set of
approved packages changes.

1. **PB hook** fires on create/update/delete of an approved record. It
   POSTs to GitHub:
   ```
   POST https://api.github.com/repos/{OWNER}/{REPO}/dispatches
   Authorization: Bearer ${GITHUB_PAT}
   { "event_type": "rebuild-site" }
   ```
   `GITHUB_PAT` is read from PB's environment at runtime (fine-grained PAT
   scoped to this repo with `Actions: write`).

2. **GitHub workflow** at `.github/workflows/rebuild.yml` listens for
   `repository_dispatch: types: [rebuild-site]`. It:
   - Checks out the repo.
   - `npm ci && npm run build` in `web/`, with `PUBLIC_PB_URL` pointing at
     prod PB so build-time fetches see the latest approved set.
   - Builds and pushes a container image including `pocketbase/` (binary +
     migrations + hooks) and the freshly built `pb_public/`.
   - Triggers the user's deploy mechanism.
   - Uses `concurrency: { group: rebuild-site, cancel-in-progress: true }`
     so bursty approvals collapse into a single build.

3. **Prod PB** must be reachable from GitHub-hosted runners (public URL).
   The build only reads approved records, which are publicly listable, so no
   credentials are needed for the fetch.

`pb_data/` (SQLite + uploads) lives outside the container image as a
mounted volume, so rebuilds don't disturb runtime data.

## v1 scope decisions

- Single GitHub repo per package row; `github_url` is the unique key.
- Display name is `owner/repo`, derived from the URL — not user-editable.
- One submitter per package, auto-set from Google profile (no co-authors).
- Curated tag taxonomy (no free-form).
- Admin moderation before public listing; admins use the built-in PB admin UI.
- GitHub metadata auto-fetched on submit and refreshed nightly.
- Listing and detail pages are SSG; PB hook triggers a GitHub Action rebuild
  on approve/edit/delete (~1–3 min lag).

## Out of scope for v1

- Comments, ratings, favourites.
- Multi-author / co-maintainer support.
- Custom moderation UI (PB admin is enough).
- Pagination beyond a simple "load more".
